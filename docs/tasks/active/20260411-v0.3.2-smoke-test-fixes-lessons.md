# v0.3.2 smoke test fixes — lessons

Paired lessons file for
[`20260411-v0.3.2-smoke-test-fixes-todo.md`](./20260411-v0.3.2-smoke-test-fixes-todo.md).
Captures what worked, what bit, and what to watch for next time.

## What the real-world `form.docx` smoke test surfaced

One 1.2 MB Korean business form (46 tables, inline images, headers
with `{pageNumber}`, cells with backgrounds, horizontally merged
cells, and 500+ `w:val="0"` style overrides) revealed four
preexisting editor bugs that had passed every unit, integration,
browser, and CI lane cleanly. None of them were DOCX-import
regressions — they were latent bugs that had simply never been
exercised by a real document.

The lesson is not "write more tests" (we did — 12 tests weren't
enough), it is **run the editor against at least one large
real-world document before declaring a release green**. The
pre-release docker image, the production endpoint smoke test, and
even the image upload proof via `NoSuchKey` all passed. It was
loading an actual document that exposed the bugs within minutes.

## Bug A — `w:val="0"` means OFF, not ON

OOXML on/off toggles (`<w:b>`, `<w:i>`, `<w:strike>`, `<w:u>`,
`<w:highlight>`) default to **on when the element is present
without a val**. `val="0"` / `val="false"` means off — used to
override an inherited style on a specific run.

Underline and highlight had already been fixed (for
`val="none"` on underline and `val="none"` on highlight). Bold,
italic, and strikethrough were still checking element presence
only, so 1531 runs in `form.docx` that were *explicitly clearing*
inherited bold/italic/strikethrough all flipped back on during
import.

**Fix pattern:** extract an `isToggleOn(el)` helper and route
every two-state element through it. Done once, it generalizes to
any future toggles. The tests now include a `"0"` / `"false"`
matrix for each, so the next toggle added gets the regression
guard for free if the author copies the pattern.

## Bug B — `getElementsByTagNameNS` is recursive

A DOM method that looks like an array accessor is actually a
deep-tree search. The table row walk had been rewritten to use
direct-child iteration specifically to avoid nested-table
bleedover, but the `gridCol` lookup was left as a recursive
`getElementsByTagNameNS` call — asymmetry that only showed up
when an outer table wrapped nested tables.

**Lesson:** `getElementsByTagName*` and `querySelectorAll` walk
the whole subtree. When one part of a traversal uses direct-child
iteration for correctness, the other parts of the same traversal
must too. Matching patterns beat matching comments — during
review, "also direct-child" only matters if it's observable at
the call site.

**Fix pattern:** a `findDirectChild(parent, localName)` helper
for node-level lookups. Used once here for `tblGrid`, available
for any future direct-child walk.

## Bug C — Paint order is a z-stack, not a pipeline

The editor's render used to do "highlights → text (including
tables)" on the assumption that tables *are* text. But
`renderTable` internally painted cell backgrounds as opaque
fillRects **before** the cell text, which meant the table body
drew cell bg *after* the selection highlight layer, covering
it. The theme's `selectionColor` is `rgba(..., 0.3)` — explicitly
designed to overlay on top of backgrounds — so the fix was to
make the draw order match that intent.

**Fix pattern:** split a composite render into "backgrounds"
and "content" passes, run backgrounds first so any higher
layers (selections, peer cursors, search highlights) land on
top of them, and keep `renderTable` as a facade for
single-pass callers (tests, exports, anything that doesn't
interleave with page-level highlights).

**Known limitation carried forward:** inline run backgrounds
(`style.backgroundColor` on `Inline`) still draw inside the
text pass, so selecting over a highlighted run still loses the
selection color. Same root cause, bigger refactor — needs
inline bg to be hoisted into the pre-pass layer across all
text drawing, not just tables. Deferred.

## Bug D — The cells array is keyed by grid column, not by tc

The model layer's `Doc.mergeCells` establishes the contract:
`row.cells.length === tableGrid.length`, with covered positions
marked `colSpan === 0`. Every consumer — layout, renderer, click
handler, exporter — indexes by grid column and uses the
placeholder sentinel to find the owner cell.

The DOCX importer honored this convention for `<w:vMerge/>` but
not for `<w:gridSpan/>`. Horizontally merged rows were shorter
than `numCols`, so `cells[c]` for the "right half" of a merged
cell was `undefined`, not a placeholder. The click handler's
owner-finding fallback was gated on `cell?.colSpan === 0`, which
silently skipped `undefined`, so the cursor couldn't land inside
the merged cell.

**Lesson:** a data-model contract is only as strong as the
producers that emit data into it. When the downstream indexer
assumes length N, every upstream producer has to emit N entries.
The fix here is trivial (pad the placeholders) but finding it
required tracing the click → click handler → data model path
end to end, because the symptom ("clicking doesn't work") was
several layers away from the root cause ("importer pushed
fewer entries than grid columns").

**Fix pattern:** extract a `makeCoveredCell()` helper used by
both merge branches so the invariant is visually obvious at the
push site. Tests that assert on `row.cells.length === numCols`
plus `colSpan === 0` on the covered positions lock the contract.

## Cross-cutting meta-lessons

- **Rename task files via `mv` + commit.** The first pass of
  this work called the file `20260411-docx-import-bugs-todo.md`,
  which became wrong as soon as Bug C (not an import bug) was
  added. Renaming early kept the commit that broadened scope
  atomic with the new content and avoided a trailing "rename"
  commit.
- **Write the failing test before the fix.** Every bug in this
  PR went red → fix → green, and every test was new. None were
  speculative; each reproduces a specific input from `form.docx`
  or a directly analogous minimal case. This made the PR review
  faster because the reviewer could see the invariant being
  protected alongside the code change.
- **Prefer static regression guards over heavy integration
  tests.** The CodeRabbit review asked for a `DocCanvas.render()`
  canvas-spy test for Bug C. The minimal test would have
  required a full canvas shim, paginated layout fixture, and
  selection setup. Instead, a static import-scan test locks
  down the specific regression (doc-canvas drifting back to
  the `renderTable` facade) with two grep-like assertions. The
  narrower test catches the narrower bug.
- **Codecov should not gate merge.** PR #117 failed
  `codecov/patch` because the new tests didn't exercise every
  branch of the pre-existing `DEFAULT_CELL_STYLE` object
  literal. Coverage is a trailing indicator; it reports on what
  existed, not on what's good. Added `codecov.yml` with
  `informational: true` so coverage posts in the PR comment but
  never blocks.
- **Post-release smoke tests must use real content, not
  synthetic.** The v0.3.2 release passed `/auth/me`, verified
  S3 credentials via a `NoSuchKey` endpoint probe, and confirmed
  Docker publish. None of that exercised the actual editor
  surface. Import-a-real-document should be a standard smoke
  test step for any release that touches the editor or import
  path.

## Follow-ups

- Lift inline run backgrounds out of the text pass into the
  same pre-pass layer as table cell backgrounds (blocker for
  full Bug C parity on highlighted runs).
- Add `form.docx` (or a trimmed proxy) as an optional
  end-to-end fixture for the browser harness, so the same
  bugs cannot silently recur on future releases.
- Once Bug A/B/D land, re-examine `docx-exporter` round-trip
  tests against the same `form.docx` sample to catch any
  export-side symmetry gaps (e.g., does exporting a padded
  placeholder produce a correct `<w:gridSpan/>`? The current
  tests say yes but they use hand-crafted input).
