# v0.3.2 smoke test fixes (v0.3.3) — todo

## Summary

Four bugs surfaced during the v0.3.2 browser smoke test against a
real-world `form.docx` (1.2 MB, 46 tables, 513 `<w:strike w:val="0"/>`
runs). All four are editor / import path only (no server, schema,
or infra impact) and will ship together in v0.3.3.

- **Bug A** — DOCX importer treats `<w:b/italic/strike w:val="0"/>`
  as ON, so explicit "off" toggles become forced "on" across the
  document.
- **Bug B** — DOCX importer's `tbl.getElementsByTagNameNS('gridCol')`
  is recursive and leaks nested-table columns into the outer
  `columnWidths`, collapsing wrapper tables to a narrow strip on
  later pages.
- **Bug C** — Editor's table cell backgrounds are drawn **after**
  the selection highlight, so selecting text inside a colored cell
  is invisible. Same file cluster as Bug B — split the table
  render pipeline into a background pre-pass.
- **Bug D** — DOCX importer does not pad horizontal merge
  (`gridSpan > 1`) placeholders into the row's cells array. The
  data-model contract is `row.cells.length === numCols` with
  covered positions marked `colSpan === 0`, which `mergeCells` in
  the model honors. Every downstream consumer — layout,
  renderer, click handler, exporter — indexes `rows[r].cells[c]`
  by grid column and relies on that contract, so an imported
  merge leaves `cells[c]` undefined for the covered columns and
  clicks on the right half of a merged cell fall through to
  nowhere.

Release cut for v0.3.3 is out of scope for this task file.
See `docs/design/project_wafflebase_release_policy.md` — v0.3.3
is expected to be a routine image bump.

## Bug A — `w:val="0"` treated as ON for bold / italic / strikethrough

**File:** `packages/docs/src/import/docx-style-map.ts:21-23`

The current code checks only for element presence:

```typescript
if (getW(rPr, 'b')) style.bold = true;
if (getW(rPr, 'i')) style.italic = true;
if (getW(rPr, 'strike')) style.strikethrough = true;
```

OOXML uses `<w:b w:val="0"/>`, `<w:i w:val="0"/>`,
`<w:strike w:val="0"/>` (and the equivalent `w:val="false"`) to
**explicitly clear** an inherited style. Missing `w:val` means ON.

The already-landed fixes for underline (bare `<w:u/>` vs
`<w:u w:val="none"/>`) and highlight (`<w:highlight w:val="none"/>`)
show the correct pattern to follow.

**Impact in `form.docx`:** 509 runs with `w:b w:val="0"`, 509 with
`w:i w:val="0"`, 513 with `w:strike w:val="0"` — all incorrectly
forced ON, which is why heavy strikethrough noise appears over
large sections of the document.

### Fix

Extract a small helper:

```typescript
/**
 * True when an on/off OOXML toggle is on. A missing val defaults
 * to on (matches Word); "0" and "false" turn it off.
 */
function isToggleOn(el: Element | null): boolean {
  if (!el) return false;
  const v = getWAttr(el, 'val');
  return v !== '0' && v !== 'false';
}
```

Replace the three presence checks with `isToggleOn(getW(rPr, ...))`.

- [ ] `<w:b w:val="false"/>` → `bold` not set

## Bug B — `tbl.getElementsByTagNameNS('w:gridCol')` leaks nested tables

**File:** `packages/docs/src/import/docx-importer.ts:212-219`

```typescript
const gridCols = tblEl.getElementsByTagNameNS(W, 'gridCol');
```

`getElementsByTagNameNS` is **recursive** and returns every
descendant `w:gridCol`, including those inside nested tables.
The `<w:tr>` walk already uses direct-child traversal
(`tblEl.childNodes` + `localName === 'tr'`) so the asymmetry is
the root cause.

**Impact in `form.docx`:** 6 top-level tables have inflated column
counts:

| idx | own cols | recursive cols | nested | result width |
|---|---|---|---|---|
| 6  | 3 | 16 | 4 nested | ~19% |
| 7  | 1 | 12 | 4 nested | ~8%  |
| 9  | 1 | 12 | 4 nested | ~8%  |
| 11 | 1 | 12 | 4 nested | ~8%  |
| 13 | 1 | 12 | 4 nested | ~8%  |
| 15 | 1 | 12 | 4 nested | ~8%  |
| 17 | 1 | 5  | 2 nested | ~20% |

All of these are "별첨 n-k 인적사항" forms — the 1-column outer
table collapses to ~8% of the content width with every row
wrapping vertically in a narrow strip.

### Fix

Walk direct children to find the first `<w:tblGrid>`, then walk
its direct children for `<w:gridCol>`. Helper for clarity:

```typescript
function directChildByLocalName(parent: Element, name: string): Element | null {
  for (let i = 0; i < parent.childNodes.length; i++) {
    const n = parent.childNodes[i];
    if (n.nodeType === 1 && (n as Element).localName === name) {
      return n as Element;
    }
  }
  return null;
}
```

Use it to find `<w:tblGrid>` directly under `<w:tbl>`, then iterate
its direct-child `<w:gridCol>` elements.

- [ ] **Regression:** 1-column outer table containing a nested

## Bug C — Selection highlight hidden under table cell background

**Files:**
- `packages/docs/src/view/table-renderer.ts` — split render into passes
- `packages/docs/src/view/doc-canvas.ts` — insert background pre-pass
  before selection drawing

The editor's page render lays down highlights (search, peer cursor,
local selection) and then draws text, including tables. `renderTable`
internally draws cell backgrounds first, then text, then borders.
Because the table render runs **after** the selection, the opaque
cell background fillRects cover the translucent blue selection
overlay — selecting text inside a colored cell looks identical to
not selecting it.

`Theme.selectionColor` is `rgba(66, 133, 244, 0.3)` (30% alpha), so
it *is* designed to overlay on top of backgrounds. Moving the cell
backgrounds earlier in the draw order preserves that intent without
inverting the order for text (text still draws on top of the
selection for readability).

### Fix

Split `renderTable` into two public functions:

- `renderTableBackgrounds(...)` — just the per-cell fillRect pass
- `renderTableContent(...)` — text (including inline run backgrounds),
  list markers, borders

Keep `renderTable` as a facade that calls both in sequence so
single-pass callers / tests don't have to change.

In `doc-canvas.ts`:

1. Add a pre-pass before the selection block that walks the
   current page's lines and calls `renderTableBackgrounds` for
   each visible table block.
2. In the main text loop, call `renderTableContent` instead of
   `renderTable` — the background is already drawn.
3. Extract `computeTableRangeForPageLine` +
   `collectTableRenderRanges` helpers so both passes share the
   row-range logic (forward peek over consecutive PageLines,
   backward extension over rowSpan owners from previous pages).

### Known limitation (out of scope)

Inline run backgrounds (`style.backgroundColor` on `Inline`) have
the same class of bug: they are drawn from inside the text pass,
which runs after the selection. Selecting over a highlighted run
still loses the selection color. Fixing this requires lifting
inline bg out of the text pass into the same pre-pass layer — a
bigger refactor because the text pass draws inline bg + text
together per run. Tracked as a follow-up.

## Bug D — Horizontal merge placeholders missing from imported rows

**File:** `packages/docs/src/import/docx-importer.ts:273-327`

Symptom: in a form imported from DOCX with a row like
`[A, B(gridSpan=4)]`, clicking the right half of `B` does
nothing — the cursor does not land inside the merged cell.

The data-model contract established by `Doc.mergeCells` in
`packages/docs/src/model/document.ts` is that a row's `cells`
array has one entry per grid column, with merged-covered
positions marked `colSpan === 0`. Every consumer reaches into
the row by grid column:

- `table-layout.ts` iterates `for (let c = 0; c < numCols; c++)`
  and uses `colSpan === 0` to push a placeholder layout cell
- `table-renderer.ts` indexes `rows[r].cells[c]` by grid column
- `text-editor.ts` click handler (`resolveTableCellClick`)
  finds the owner by stepping back when `cell?.colSpan === 0`
- `docx-exporter.ts` skips cells whose `colSpan === 0` when
  writing `<w:tc>`s out

The importer honored this contract only for vertical merges
(`<w:vMerge/>` continue pushed one covered cell). For horizontal
merges it pushed a single owner with `colSpan: N` and then
advanced `colIdx += N` **without** emitting the N − 1 covered
placeholders. Result: the cells array was shorter than
`numCols`, so the click handler's owner-finding fallback (gated
on `cell?.colSpan === 0`) silently skipped the merged range
because `cell` was `undefined`, not a placeholder.

Edge case: a `<w:tc>` can have both `<w:gridSpan/>` **and**
`<w:vMerge/>` (a vertically merged cell that also spans
multiple grid columns). The old vMerge branch pushed only one
covered cell regardless of the gridSpan, which drifted the row
the same way.

### Fix

Extract a `makeCoveredCell()` helper at module scope. Use it in
both merge branches:

- `vMerge === 'continue'`: push one covered cell per grid column
  the tc spans (`for (let s = 0; s < colSpan; s++)`).
- Normal cell with `gridSpan > 1`: after pushing the owner,
  pad `(colSpan − 1)` placeholders.

No change to `colIdx += colSpan`. The model, layout, renderer,
click handler, and exporter all stay unchanged because the
input they see now matches the established contract.
